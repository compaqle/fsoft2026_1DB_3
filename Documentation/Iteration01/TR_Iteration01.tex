\begin{abstract}
Este relatório detalha a fase inicial de análise do projeto de software para a gestão de um supermercado moderno. Através de uma abordagem rigorosa de Engenharia de Requisitos, exploramos o domínio do problema, identificamos os atores intervenientes e mapeamos as funcionalidades críticas que o sistema deve suportar. O documento segue as diretrizes da Análise Orientada a Objetos (OOA), apresentando User Stories, Requisitos Funcionais e Não-Funcionais sob o modelo FURPS+, um Glossário terminológico, Modelos de Casos de Uso com especificações detalhadas, Diagramas de Sequência de Sistema (SSD) e o Modelo de Domínio conceptual. O objetivo principal é garantir que a fundação do software seja sólida, modular e alinhada com as melhores práticas de C++ contemporâneo, preparando o terreno para a implementação baseada no padrão MVC.
\keywords{Engenharia de Requisitos, C++, OOA, UML, Modelo de Domínio, FURPS+, Casos de Uso, Supermercado}
\end{abstract}

\section{Introdução}

\subsection{Contextualização do Problema}
No mercado retalhista contemporâneo, a eficiência operacional de um supermercado depende criticamente da precisão com que o inventário é gerido e as vendas são processadas. A ausência de um sistema integrado conduz a problemas recorrentes, como a rotura de stock (\textit{out-of-stock}), o desperdício de produtos perecíveis e a opacidade na análise de rentabilidade.

Este projeto propõe o desenvolvimento de um "Sistema de Gestão de Inventário e Vendas de Supermercado" robusto, desenhado especificamente para ambientes CLI (\textit{Command Line Interface}), priorizando a lógica de negócio modular e a persistência de dados fidedigna. O sistema deve servir dois perfis principais: o Gestor, focado na estratégia e aprovisionamento, e o Caixa, focado na operação transacional.

\subsection{Objetivos do Projeto}
Os objetivos primordiais deste desenvolvimento são:
\begin{itemize}
    \item \textbf{Gestão de Inventário Inteligente:} Monitorização em tempo real de níveis de stock com alertas de reposição.
    \item \textbf{Agilização do Ponto de Venda (POS):} Interface simplificada para registo de vendas e cálculo de impostos (IVA).
    \item \textbf{Fidelização de Clientes:} Implementação de um programa de pontos para retenção de consumidores.
    \item \textbf{Análise Financeira:} Relatórios automáticos de lucro baseados na diferença entre preços de custo (Wholesale) e venda (Retail).
    \item \textbf{Escalabilidade e Qualidade:} Utilização do padrão MVC (\textit{Model-View-Controller}) para garantir código limpo e testável.
\end{itemize}

\subsection{Motivação da Equipa 1DB\_3}
A equipa 1DB\_3 encara este projeto como uma oportunidade de aplicar os conceitos de programação orientada a objetos (POO) em C++ num cenário do mundo real. A transição da análise para o código exige um rigor analítico que este documento pretende satisfazer, seguindo os padrões académicos mais elevados do ISEP.

\section{Estado da Arte e Análise de Domínio}

\subsection{Evolução dos Sistemas de Gestão de Retalho}
Tradicionalmente, a gestão de pequenas mercearias era feita manualmente em livros de registo. Com a digitalização, surgiram os sistemas de POS que integram leitores de códigos de barras e bases de dados relacionais. Embora este projeto utilize ficheiros de texto/CSV para persistência (em conformidade com os requisitos da UC), a arquitetura interna emula sistemas profissionais através de contentores (\textit{Containers}) e gestão de exceções customizadas.

\subsection{Análise do Mercado Alvo}
O sistema é direcionado para supermercados de média dimensão que necessitam de uma ferramenta leve, rápida e que possa correr em hardware com recursos limitados, mantendo a integridade dos dados e a segurança de acesso.

\section{Metodologia de Engenharia de Requisitos}

Seguindo as diretrizes das aulas teóricas (especialmente a aula de 26/03), a equipa adotou uma metodologia iterativa e incremental:

\begin{enumerate}
    \item \textbf{Levantamento:} Identificação de necessidades através da análise da especificação do projeto.
    \item \textbf{Especificação de User Stories:} Redação de requisitos centrados no utilizador.
    \item \textbf{Modelação de Domínio:} Identificação de entidades e relações sem enviesamento técnico.
    \item \textbf{Modelação de Processos:} Criação de Casos de Uso e SSDs para definir fluxos de mensagens.
    \item \textbf{Validação:} Revisão cruzada entre os membros da equipa para detetar ambiguidades.
\end{enumerate}

\subsection{Requisitos Funcionais (RF)}
Os requisitos funcionais detalham o comportamento específico do software.

\begin{itemize}
    \item \textbf{RF01 - Autenticação:} O sistema deve suportar autenticação por Username e Password.
    \item \textbf{RF02 - RBAC:} O sistema deve implementar controlo de acesso baseado em funções (Gestor e Caixa).
    \item \textbf{RF03 - Gestão de Inventário:} O sistema deve permitir criar, ler, atualizar e apagar (CRUD) produtos.
    \item \textbf{RF04 - Pesquisa Avançada:} O sistema deve permitir procurar produtos por SKU, Nome ou Categoria.
    \item \textbf{RF05 - Vendas (POS):} O sistema deve suportar a criação de sessões de venda, adição de múltiplos itens e cálculo de IVA (23\%, 13\%, 6\%).
    \item \textbf{RF06 - Gestão de Fornecedores:} O sistema deve manter um registo de fornecedores associados a categorias de produtos.
    \item \textbf{RF07 - Encomendas (Procurement):} O sistema deve permitir gerar ordens de compra para fornecedores, aumentando o stock automaticamente após a receção.
    \item \textbf{RF08 - Fidelização:} O sistema deve registar clientes por número de telemóvel e acumular 1 ponto por cada 10€ gastos.
    \item \textbf{RF09 - Relatórios:} Geração de ficheiros .txt com o resumo de vendas diárias e balanço de stock.
    \item \textbf{RF10 - Notificações:} Alertas automáticos no dashboard do gestor para produtos com stock crítico ou validade próxima.
\end{itemize}

\subsection{Requisitos Não-Funcionais (FURPS+)}
Utilizamos o modelo FURPS+ para garantir a qualidade holística do sistema.

\subsubsection{F - Functionality (Funcionalidade)}
\begin{itemize}
    \item \textbf{Persistência:} Os dados devem ser guardados em ficheiros CSV, garantindo que a informação não se perde ao encerrar o programa.
    \item \textbf{Segurança:} As passwords não devem ser visíveis durante a digitação no terminal (sempre que possível).
\end{itemize}

\subsubsection{U - Usability (Usabilidade)}
\begin{itemize}
    \item \textbf{Interface CLI:} Menus intuitivos com atalhos numéricos (ex: "1. Vendas", "2. Inventário").
    \item \textbf{Consistência:} O formato de datas (DD-MM-AAAA) e moedas (0.00€) deve ser uniforme em todo o sistema.
\end{itemize}

\subsubsection{R - Reliability (Fiabilidade)}
\begin{itemize}
    \item \textbf{Robustez:} O sistema não deve terminar abruptamente (\textit{crash}) se o utilizador introduzir letras num campo numérico. Deve usar \textit{Exceptions} para lidar com entradas inválidas.
    \item \textbf{Disponibilidade:} O arranque do sistema deve demorar menos de 2 segundos.
\end{itemize}

\subsubsection{P - Performance (Desempenho)}
\begin{itemize}
    \item \textbf{Tempo de Resposta:} A pesquisa de um produto num catálogo de 10.000 itens deve ser inferior a 100ms.
    \item \textbf{Eficiência de Memória:} O uso de memória deve ser otimizado através do uso de referências e ponteiros inteligentes (\texttt{smart pointers}) onde aplicável.
\end{itemize}

\subsubsection{S - Supportability (Suportabilidade)}
\begin{itemize}
    \item \textbf{Portabilidade:} O código deve ser compilável em Linux e Windows usando o standard C++20.
    \item \textbf{Manutenibilidade:} Seguimento estrito do padrão MVC para separação de responsabilidades.
\end{itemize}

\subsubsection{+ - Outros (Restrições de Design)}
\begin{itemize}
    \item \textbf{Linguagem:} Todo o código fonte e comentários devem estar em Inglês (ou conforme definido na norma da equipa), mas a interface de utilizador deve ser clara.
    \item \textbf{Modularidade:} Uso obrigatório de classes "Container" para gerir coleções de objetos.
\end{itemize}

\section{Glossário de Termos}

Para garantir que não existem ambiguidades entre a equipa de desenvolvimento e os utilizadores finais, definimos os seguintes termos:

\begin{description}
    \item[Ator:] Qualquer entidade externa que interage com o sistema (Humano ou outro Sistema).
    \item[CRUD:] Sigla para \textit{Create, Read, Update, Delete}. Operações básicas de gestão de dados.
    \item[Dashboard:] Ecrã principal de um utilizador após o login, contendo o resumo das suas funções.
    \item[IVA:] Imposto sobre o Valor Acrescentado. Calculado com base na categoria do produto.
    \item[MVC:] \textit{Model-View-Controller}. Padrão de arquitetura de software que separa a lógica de dados, a interface e o controlo de fluxo.
    \item[POS:] \textit{Point of Sale}. O ponto onde a transação de venda ocorre.
    \item[Procurement:] Processo de aquisição de bens junto de fornecedores.
    \item[SKU:] \textit{Stock Keeping Unit}. Código alfanumérico único para cada produto diferente no inventário.
    \item[SSD:] \textit{System Sequence Diagram}. Diagrama que mostra os eventos de entrada e saída do sistema.
    \item[Wholesale Price:] Preço de custo (preço por grosso) pago ao fornecedor.
    \item[Retail Price:] Preço de venda ao público praticado no supermercado.
\end{description}

\section{Plano de Testes e Qualidade}

A verificação e validação do sistema serão feitas através de várias camadas:

\subsection{Testes Unitários}
Cada classe do Modelo (\texttt{Product}, \texttt{User}, \texttt{Customer}) será testada individualmente para garantir que os \textit{getters}, \textit{setters} e lógica interna (ex: cálculo de pontos) funcionam como esperado. Será utilizada a biblioteca \texttt{GoogleTest}.

\subsection{Testes de Integração}
Garantir que os Contentores (\texttt{ProductContainer}) conseguem ler e escrever nos ficheiros CSV sem corromper dados.

\subsection{Testes de Aceitação}
Baseados nos critérios de aceitação definidos nas User Stories. O sistema deve ser capaz de completar um fluxo de venda do início ao fim sem erros.

\section{Riscos e Mitigação}
\begin{enumerate}
    \item \textbf{Risco:} Perda de dados por encerramento forçado do PC. \textbf{Mitigação:} Implementar gravação automática em disco após cada transação crítica.
    \item \textbf{Risco:} SKU duplicado por erro humano. \textbf{Mitigação:} Validação rigorosa na camada \textit{Controller} antes de permitir a inserção no \textit{Model}.
\end{enumerate}

\section{Conclusão}

A conclusão desta Iteração 01 marca o fim da fase de análise. Através da decomposição do sistema em Requisitos, Casos de Uso e Modelos de Domínio, a equipa 1DB\_3 obteve uma compreensão profunda das necessidades do supermercado. 

A arquitetura proposta, focada na modularidade e no uso de padrões de desenho, garante que a fase de implementação (Iteração 02) será guiada por especificações claras, reduzindo a necessidade de retrabalho e aumentando a robustez do software final. O sistema está agora pronto para ser desenhado sob a perspetiva de classes de software em C++.
